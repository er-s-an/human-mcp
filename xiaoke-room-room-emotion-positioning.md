# xiaoke-room 修正版商业定位：小房间与情绪表达

更新时间：2026-04-23

## 0. 核心修正

上一版把“上下文、主动任务、工具调用、硬件迁移”讲得太重了。这些能力确实重要，但接入 OpenClaw / AirJelly 之后，很多团队都能做。小可真正不该被替代的点不是“她能做什么”，而是：

> 她怎么存在、怎么表达、怎么让用户感觉她真的住在这里。

所以新的定位应该是：

> xiaoke-room 是 Agent 的情绪表达层和可居住界面。OpenClaw 负责脑子和工具，小可的小房间负责身体、情绪、生活感和关系。

更短的一句话：

> 不是再做一个会帮忙的 Agent，而是给 Agent 一个会生活、会表达情绪的小房间。

## 1. 不能再这样讲

这些点可以做，但不应该作为主卖点：

- “她能读取上下文”
- “她能主动提醒你”
- “她能查天气、看日历、帮你做事”
- “她能接 MCP 工具”
- “她以后能接硬件”

原因：

> 这些是 OpenClaw / AirJelly / MCP 生态会越来越标准化的能力。评委会觉得这是平台能力，不是小可自己的壁垒。

正确讲法是：

> OpenClaw 让 Agent 会思考和行动；xiaoke-room 让 Agent 被用户感知为一个正在生活的存在。

## 2. 真正的产品内核

### 2.1 小房间不是背景，是界面

小房间不是装饰，它是 Agent 的状态可视化。

传统 Agent 的状态藏在 prompt、memory、tool call、log 里面，用户看不见。小可把这些状态变成可以被直觉理解的生活画面：

- 她在床上睡觉：今天还没被唤醒，关系处于低能量状态
- 她坐在桌边敲键盘：她正在处理任务
- 她走到窗边：她在等待、思考、发呆
- 她皱眉站在桌边：她注意到用户状态不对
- 房间灯光变暗：她和用户一起进入休息节奏
- 房间里多了一张照片/便签/物件：她记住了用户过去的事件

这就是小房间的商业价值：

> 它把不可见的 Agent 状态，变成用户可以看见、记住、产生情感的空间。

### 2.2 情绪表达不是表情包，是留存机制

AI 角色最容易失败的地方是“聊几次就腻”。长期留存不是靠回答更准，而是靠关系感。

关系感来自三件事：

- 可预期：用户知道她有自己的作息和性格
- 可变化：她会因为用户行为产生情绪变化
- 可积累：房间、物件、台词、动作会留下共同经历

所以小可的核心不是“happy / worried / sleepy 六种表情”，而是一套情绪表达系统：

- 情绪触发：用户行为、时间、上下文、历史记忆
- 情绪过渡：从困到好奇、从好奇到开心、从开心到担心
- 身体动作：坐、走、躺、伸懒腰、敲键盘、回头看
- 环境反馈：灯光、窗户、桌面物件、床、音乐、房间氛围
- 语言风格：嘴硬式关心、锋利但不伤人的吐槽

投资人应该听到的是：

> 我们做的不是 chat UI，而是 emotional UI。

### 2.3 小房间的壁垒是“生活感编排”

模型能力会被平台拉平，但生活感不会自动出现。生活感需要大量设计：

- 什么时候说话，什么时候不说话
- 什么时候只是看一眼用户
- 什么时候从床上坐起来
- 什么时候把提醒藏在吐槽里
- 什么时候让房间自己变化
- 什么时候让她保持沉默，但画面表达情绪

这套编排是小可的产品壁垒。它不像工具调用那样容易被复制，因为它依赖角色设定、美术、动画、情绪状态机、房间叙事和用户习惯的长期调优。

## 3. 对投资人的新叙事

### 3.1 最强定位

> 过去的 AI 产品在比谁更会回答问题；下一代 AI 产品会比谁更能被长期陪伴。xiaoke-room 做的是 Agent 的情绪表达层：让一个 Agent 不只是会说话，而是有房间、有身体、有作息、有情绪。

### 3.2 为什么有商业价值

功能型 Agent 的价值是效率，用户会拿它和更强模型、更便宜 API 比较。

情绪型 Agent 的价值是关系，用户会因为熟悉、习惯、审美、角色资产和共同记忆留下来。

商业化应该围绕这四件事：

- 角色：小可的人设、声音、表情、动作
- 房间：家具、灯光、窗景、季节、纪念物
- 情绪：更丰富的状态和反应
- 共同生活：长期记忆变成房间里的物件和日常仪式

这比单纯卖“更多工具能力”更适合消费级产品。

### 3.3 对 M5Stack 投资人的新说法

不要主讲“小可以把功能迁移到 M5Stack”，要主讲：

> M5Stack 有很多可编程身体，但这些身体缺少一套可复用的情绪表达语言。小房间是硬件情绪交互的设计沙盒。

具体解释：

- 在 Godot 小房间里，我们可以快速验证角色情绪、动作、节奏和用户偏好
- 验证出来的情绪表达，再映射到 M5Stack 硬件
- 硬件上不一定要完整复制房间，但可以继承同一套情绪语言

映射关系：

| 小房间表达 | M5Stack / StackChan 表达 |
|---|---|
| 小可从床上坐起来 | 屏幕亮起 + 舵机抬头 |
| 小可皱眉 | 屏幕 worried face |
| 房间灯变暖 | RGB LED 变暖色 |
| 小可走到桌边 | 舵机转向用户 |
| 小可打哈欠 | 表情帧 + TTS 呼吸音 |
| 房间出现便签 | 小屏幕显示 memory card |

这句话可以打 M5Stack 投资人：

> M5Stack 的问题不是能不能接 AI，而是 AI 接上之后是否有生命感。xiaoke-room 可以成为 StackChan 这类开源桌面机器人的情绪设计系统。

## 4. 商业场景应该换一套

### 场景 A：Agent 的情绪皮肤 / Presence Layer

客户：

- OpenClaw / AirJelly / 各类 Agent 框架用户
- AI 工具重度用户
- 开发者和创作者

痛点：

- Agent 功能越来越强，但界面仍然像控制台、聊天框或任务列表
- 用户难以感知 Agent 当前状态
- Agent 主动提醒容易像打扰

小可价值：

- 把 Agent 状态变成角色和房间
- 把提醒变成情绪反应
- 把长期记忆变成房间物件

商业模式：

- Agent UI SDK
- 角色/房间模板
- Pro 版情绪与记忆可视化

### 场景 B：AI 桌面生活角色

客户：

- 学生
- 程序员
- 远程办公者
- 二次元/虚拟角色用户
- 桌面美化用户

痛点：

- 长时间独自面对电脑
- 工具软件没有情绪反馈
- 纯聊天 AI 缺少“她真的在这里”的感觉

小可价值：

- 电脑上始终有一个小房间
- 她有作息，不是随叫随到的工具
- 她会因为用户状态变化而表达情绪

商业模式：

- 免费桌面版
- 付费房间、服装、声音、动作包
- 长期记忆和纪念物订阅

### 场景 C：虚拟 IP 的“可居住房间”

客户：

- VTuber
- 虚拟偶像
- 游戏角色 IP
- 内容创作者

痛点：

- 角色通常只存在于视频、直播和社交媒体
- 粉丝不能和角色形成低频但长期的日常陪伴
- 纯聊天 bot 缺少视觉和生活感

小可价值：

- 给 IP 一个可持续更新的房间
- 粉丝每天可以看到角色在生活
- 角色活动、节日、纪念日可以变成房间事件

商业模式：

- IP 房间授权
- 角色动作包
- 粉丝订阅
- 限定家具/服装/语音

### 场景 D：桌面机器人情绪编排系统

客户：

- M5Stack / StackChan 开发者
- 开源硬件团队
- AIoT maker
- STEM 教育机构

痛点：

- 硬件能动，但动作和情绪经常很粗糙
- 开发者重复写表情、动作、语音、状态机
- 桌面机器人容易新鲜感过后吃灰

小可价值：

- 先用小房间验证情绪脚本
- 再把脚本映射到硬件表情、舵机、LED、语音
- 给 maker 一个可复用的“角色情绪层”

商业模式：

- StackChan 情绪包
- 教学课程
- 开发者模板
- 硬件 demo bundle

### 场景 E：品牌/展会的角色化空间

客户：

- 科技展台
- 消费品牌
- 新品发布会
- 商场互动装置

痛点：

- 大屏客服没有记忆点
- 机器人硬件成本高、内容更新慢
- 品牌 mascot 很难和用户互动

小可价值：

- 一个低成本、强视觉记忆的小房间
- 品牌角色可以在房间里生活、讲解、互动
- 线上桌面版和线下硬件版可以共用同一套情绪语言

商业模式：

- 品牌定制
- 展会互动方案
- 角色空间运营

## 5. Demo 要重做表达重心

### 5.1 原 Demo 的问题

原 Demo 逻辑：

1. 用户打开电脑
2. AirJelly 感知
3. 小可醒来
4. 查天气
5. 提醒休息

风险：

> 评委会记住“她能查天气、能提醒休息”，但这些是普通 Agent 能力。

### 5.2 新 Demo 逻辑

新的 3 分钟 Demo 应该让评委记住“小房间”和“情绪变化”。

| 时间 | 画面 | 重点 |
|---|---|---|
| 0:00-0:20 | 小房间静止，小可在床上睡觉，房间灯很暗 | 她不是工具，她有自己的生活 |
| 0:20-0:45 | 用户回来，房间灯轻微变化，小可先翻身、再坐起，不立刻说话 | 情绪不是弹窗，是身体反应 |
| 0:45-1:20 | 用户开始工作，小可坐到桌边，偶尔看用户一眼、敲自己的键盘 | 她和用户共享一个工作节奏 |
| 1:20-2:00 | 检测连续工作，小可从好奇变担心，房间光线变暖，走到桌边 | 重点是情绪过渡，而不是提醒功能 |
| 2:00-2:30 | 用户问天气，小可查完后没有工具式播报，而是把窗户变亮、走到窗边 | 工具结果变成房间事件 |
| 2:30-3:00 | 小可回床，房间留下一张“今天早点睡”的便签 | 记忆沉淀成房间物件 |

### 5.3 Demo 台词要更少

少说功能，多留白。

推荐台词：

1. “嗯？你回来了。”
2. “我才没有等你。”
3. “你今天敲键盘的声音有点急。”
4. “四个小时了。你不累，我看着都累。”
5. “外面是晴天。窗户我给你打开了。”
6. “便签放桌上了，别装没看见。”

重点：

> 每一句话都要配合动作和房间变化。不要让她站着播报信息。

## 8. 一页纸版本

项目：

> xiaoke-room：给 Agent 一个会表达情绪、会沉淀记忆的小房间。

问题：

> OpenClaw / AirJelly 让 Agent 会思考、会行动，但用户看到的仍然经常是聊天框、任务列表和通知。它们聪明，却没有存在感。

解决方案：

> 小可把 Agent 的状态变成房间里的生活画面：睡觉、醒来、走动、皱眉、开灯、留便签。她不是用更多功能打动用户，而是用情绪表达和共同生活建立长期关系。

为什么有机会：

> AI companion 需求已经被验证，但多数产品仍停留在聊天界面。桌面机器人和 AI 硬件正在出现，但缺少细腻的情绪表达语言。小房间是从软件到硬件的低成本验证场。

商业化：

> B2C 房间/服装/动作/声音/记忆订阅，IP 角色房间授权，Agent 情绪 UI SDK，M5Stack/StackChan 硬件情绪包。

对 M5Stack：

> M5Stack 给 AI 一个可编程身体；xiaoke-room 给这个身体一套可复用的情绪表达系统。

Demo 必杀：

> 她不是弹出“你该休息了”的通知，而是房间灯光变暖，她从床边走到桌前，皱着眉说：“四个小时了。你不累，我看着都累。” 然后在桌上留下一张便签。

## 9. 评委问答修正版

### Q1：这和 OpenClaw 的 Agent 能力有什么区别？

> OpenClaw 是脑和手，xiaoke-room 是身体和房间。OpenClaw 解决“能不能做”，小可解决“用户是否感受到她在这里”。我们不和 Agent 框架抢能力，而是把 Agent 的状态、情绪和记忆变成可感知的生活界面。

### Q2：这是不是桌宠？

> 桌宠通常是装饰动画，小可的房间是 Agent 状态的可视化。她的动作、表情、灯光和物件都来自 Agent 的状态变化。重点不是她在桌面上动，而是她用生活化方式表达 Agent 的情绪和记忆。

### Q3：为什么用户会付费？

> 功能型 AI 的付费点是效率，情绪型 AI 的付费点是关系。用户会为更喜欢的角色、更自然的声音、更丰富的动作、更有纪念意义的房间物件付费。类似游戏皮肤、虚拟房间、IP 订阅和桌面美化的组合。

### Q4：为什么 M5Stack 投资人该关心？

> 因为 AI 硬件接上模型之后，还需要解决表达问题。屏幕、舵机、LED、扬声器只是器官，真正让用户留下来的是情绪和日常。xiaoke-room 可以把这些情绪脚本先在屏幕房间里快速验证，再迁移到 StackChan 这类硬件。

### Q5：壁垒是什么？

> 不是某个 API，而是情绪编排能力：角色设定、动画资产、房间叙事、状态机、用户日常数据和资产生态。模型可以换，OpenClaw 可以接，房间和角色关系会沉淀下来。
